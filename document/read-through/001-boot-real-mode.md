# 001_boot_real_mode —— 从 BIOS 到屏幕的第一行字

## 本章概览

这一章是 Cinux 操作系统旅程的真正起点。当按下电源键，BIOS 完成自检后会把磁盘的第一个扇区（MBR）加载到内存地址 `0x7C00`，然后跳转过去执行。这时候 CPU 处于**实模式**（Real Mode）—— 16 位寻址，只能访问 1MB 内存，没有任何保护机制。我们在这个荒岛上要完成的事情包括：初始化段寄存器、在屏幕上打印启动消息、从磁盘加载第二阶段引导程序（Stage2）、开启 A20 地址线、获取 VESA 显卡信息，最后设置 1024x768x32 的线性帧缓冲模式。

整个引导流程就像一条接力赛：BIOS 把棒交给 MBR，MBR 把棒交给 Stage2，Stage2 准备好显卡信息后交给后续的内核。每一环都不能出错，任何一个环节失败都会导致系统黑屏或者停机。

说实话，写这一章的时候我踩了不少坑。最离谱的是一开始 `print_string` 怎么都不输出，但单字符 `int 0x10` 又正常，折腾半天才发现是 CS 和 DS 不匹配的问题。后来又踩了栈踩内存、MBR 超过 512 字节、BIOS 污染寄存器等等坑。这些我都会在后续的代码讲解里详细说，你看了就不会再踩。

关键设计决策一览：
- 使用 BIOS INT 0x13 AH=0x42 扩展读取（LBA 寻址）而非传统的 CHS 方式，支持大容量磁盘
- MBR 只负责加载 Stage2，不包含复杂的初始化逻辑，保持 512 字节限制
- Stage2 负责所有硬件初始化（A20、VESA），为进入保护模式做准备
- 使用 VBE 2.0 标准获取显卡信息，选择 0x118 模式（1024x768x32）作为默认图形模式
- 将帧缓冲信息打包到固定内存地址 `0x6400`，方便后续内核代码访问

与同类 OS 项目的设计对比：Linux 早期的 bootsect.S 和 setup.S 采用了类似的两阶段引导设计，但 Linux 直接使用 8086 汇编（Intel 语法），而 Cinux 使用 GNU AS（AT&T 语法）并且从一开始就规划了更清晰的模块划分。xv6 则简化了引导流程，直接使用 GRUB，跳过了这些底层细节。

## 关键代码精讲

### MBR 入口与段初始化

当 BIOS 把控制权交给我们时，CS:IP 可能不是 0x0000:0x7C00，因为 BIOS 可能用不同的段值加载 MBR。所以我们做的第一件事就是用远跳转 `ljmp $0, $real_start` 来规范化 CS 寄存器。这个跳转会同时设置 CS 和 IP，确保跳转后 CS=0，IP=real_start 的偏移。

接下来是段寄存器的清零操作。实模式下的内存地址计算方式是 `段×16 + 偏移`，把所有段寄存器设为 0 后，地址就等于纯偏移，这样内存布局更直观。这里有个小技巧：用 `xorw %ax, %ax` 把 AX 清零，然后把 AX 复制到各个段寄存器，比直接用 `movw $0, %ax` 少一个字节（XOR 指令编码更短）。

这里我要重点说一个我踩过的坑。一开始我的 `print_string` 函数怎么都不输出，但单字符的 `int 0x10` 又正常。折腾了半天才发现问题：字符串地址是按 CS 计算的，但 `lodsb` 指令用的是 DS:SI。如果 CS ≠ DS，那读到的地址就完全错了！解决方法很简单：确保 DS = CS。在 MBR 入口处我们把所有段寄存器都清零（包括 DS），CS 也被 `ljmp` 设为 0，这样 CS=DS=0，就不会出问题了。

栈的设置也很重要。我们把栈顶 SP 设为 `0x7000`，栈向下增长。这个位置在 MBR 下方，不会覆盖 MBR 代码本身。顺便说一句，栈指针的设置必须在启用中断之前完成，否则如果有中断发生，CPU 会尝试从错误的栈地址读取数据，直接就崩溃了。

但这里又有个坑！我一开始把栈放在 `0x7B00`（MBR 附近），结果发现有时候 print 之后程序就飞了，BIOS 调用后也崩。后来才想明白：BIOS 也会用栈，你的函数也在 push，如果栈位置选得不好，会覆盖关键内存。解决方法是把栈放到更远的地方，比如 `0x7000`，这样就不会和 MBR 区域冲突了。

启动盘号的保存也是个细节。BIOS 把启动盘号放在 DL 寄存器里，我们把它保存到内存变量 `boot_drive` 中，因为后续调用 BIOS 磁盘中断时需要恢复这个值。如果不保存，等到调用 `load_stage2` 时 DL 可能已经被其他代码覆盖了。

⚠️ 千万注意：MBR 只有 512 字节！我犯过一个很蠢的错误：把 `print_string` 函数放到 `common/serial.S`，然后在 MBR 里 call 它。结果放 MBR 里正常，放到 common.S 直接就飞了。后来才想通：BIOS 只加载 512 字节，`mbr.S + serial.S` 一起链接后 `.text` 段超过 512 字节，后半部分根本没加载到内存，call 跳到未加载的内存区域当然就崩了。正确的架构是：MBR 极简（<512B），不要链接 common.S；Stage2 包含所有功能。

### print_string：字符输出到屏幕

`print_string` 函数是整个引导过程中唯一的调试输出手段，它的实现很直接但有几个细节值得注意。函数使用 `lodsb` 指令从 DS:SI 读取一个字节到 AL，然后 SI 自动加 1。这个指令是专门为字符串操作设计的，比手动用 `movb` 加 `incw` 更高效。

BIOS INT 0x10 AH=0x0E 是「teletype 输出」函数，它会在当前光标位置打印 AL 中的字符，并自动处理特殊字符。`\r` 会把光标移到行首，`\n` 会向下移动一行。我们用 `\r\n` 组合来实现标准的换行。

但这里有个超级隐蔽的坑：BIOS 中断不是函数调用！它会改寄存器，比如 DS/ES、FLAGS、BX/SI 等。如果你的函数调用 BIOS 后不恢复这些寄存器，下一个函数就炸了。我一开始就是这样：print 之后正常，下一个函数就崩。解决方法是每个函数都必须保存所有寄存器（或者至少明确哪些寄存器能改、哪些必须保存）。在我们的代码里，`print_string` 函数会在入口保存 AX、BX、SI，退出时恢复，这样就不会影响调用者了。

⚠️ 注意：字符串必须以 `\0`（null 终结符）结尾，函数会检查 AL 是否为 0 来判断是否到达字符串末尾。如果忘记加终结符，函数会继续读取内存直到遇到 0，可能打印一堆乱码。

### DAP 结构与扩展磁盘读取

传统的 BIOS INT 0x13 AH=0x02 使用 CHS（柱面-磁头-扇区）寻址，只能访问 8GB 以下的磁盘。现代磁盘动辄几百 GB，必须用 LBA（Logical Block Addressing）方式。INT 0x13 AH=0x42 是扩展读取函数，它接受一个 DAP（Disk Address Packet）结构作为参数。

我们在 `0x7B00` 处构造 DAP 结构，这个位置刚好在栈（`0x7C00`）下方，不会被栈操作覆盖。DAP 的 `size` 字段必须是 `0x10`，这是 BIOS 识别 DAP 结构的标志。`sectors` 字段设为 15，因为 Stage2 最大占用 15 个扇区。`offset` 和 `segment` 字段组合成目标缓冲区地址：`0x0000:0x8000 = 0x8000`。

LBA 从 1 开始（0 是 MBR），所以 Stage2 的起始 LBA 是 1。调用时把 DL 设为启动盘号，然后执行 `int $0x13`。返回时 Carry Flag（CF）置位表示失败，我们用 `jc`（jump if carry）检查。如果失败，打印错误消息后停机。

说实话，磁盘读取这一步最容易出问题。如果 DAP 结构填写错误，或者 LBA 不对，要么读不到数据，要么读到了错误的位置。用 QEMU 的 monitor 命令 `xp /512x 0x8000` 可以查看内存内容，验证 Stage2 是否正确加载。

### Stage2 入口与环境重初始化

Stage2 被 MBR 跳转过来时，我们知道 CS=0（因为 MBR 用 `ljmp $0, $0x8000` 跳转），但其他段寄存器可能不干净。虽然 MBR 已经清零过一次，但为了保险起见，Stage2 入口处重新清零所有段寄存器。这点冗余换来的是更可靠的代码。

栈的设置需要特别注意。Stage2 从 `0x8000` 开始，我们把栈设为 `0x9000:0xFFFE`，这样栈向下增长，不会覆盖 Stage2 代码。这里有个小技巧：`0x900 << 4 = 0x9000`，栈顶 `0xFFFE` 使得第一个 push 会写到 `0x9FFE`，刚好在 Stage2 区域下方。

打印 "Stage2 OK" 消息标志着 Stage2 成功执行。如果这时候 QEMU 窗口出现了这行字，说明 MBR 到 Stage2 的跳转成功了。这个简单的验证在调试时非常有用 —— 如果看不到这行字，问题就在 MBR 或磁盘读取。

### A20 地址线的开启

在早期的 8086 CPU 中，地址线只有 20 根，能寻址 1MB 内存。当程序访问超过 1MB 的地址时，地址会回绕到 0（比如 `0xFFFF0 + 0x10 = 0x00000`）。这在当时是个 bug，但后来有些程序依赖这个特性。286 及以后的 CPU 有更多地址线，为了兼容老程序，默认情况下第 21 根地址线（A20）是被禁用的。

我们需要显式开启 A20 才能访问超过 1MB 的内存，这对后续进入保护模式很重要。我们使用 BIOS INT 0x15 AX=0x2401 来开启 A20，这是最简单的方法。返回时 AH=0 表示成功，AL 包含 A20 门状态（1=已开启，0=已关闭）。

如果 BIOS 方法失败（CF 置位或 AH 非 0），我们直接 panic。在实际产品中可能需要尝试其他方法（键盘控制器、快速 A20 门），但对于我们的教程，一个方法就够了。大部分现代 BIOS 都支持这个中断，失败的情况很少见。

### VESA 控制器信息获取

VESA BIOS Extensions（VBE）是 BIOS 提供的一组接口，用来查询和设置显卡模式。第一步是获取控制器信息，调用 INT 0x10 AX=0x4F00，ES:DI 指向一个 512 字节的缓冲区（VbeInfoBlock）。

这里有个关键的细节：在调用前，我们需要把缓冲区的签名设为 `"VBE2"`。这告诉 BIOS 我们想要 VBE 2.0 或更高版本的信息。如果签名是 `"VBE"`，BIOS 只返回 VBE 1.2 的信息，可能不包含我们需要的模式。

VbeInfoBlock 包含很多信息，但我们只关心它证明了 VBE 的存在。返回时 AL=0x4F 表示支持 VBE，AH=0 表示调用成功。如果这两个检查有一个失败，我们就 panic，因为后续的图形模式初始化完全依赖 VBE。

### VESA 模式信息获取

获取控制器信息后，下一步是查询具体模式的信息。我们要用的是模式 `0x118`（1024x768x32），调用 INT 0x10 AX=0x4F01 CX=0x0118，ES:DI 指向一个 256 字节的缓冲区（ModeInfoBlock）。

ModeInfoBlock 包含模式的详细信息，我们最关心的是：
- `PhysBasePtr`（偏移 0x28，4 字节）：帧缓冲的物理地址
- `BytesPerScanLine`（偏移 0x10，2 字节）：一行占用的字节数（pitch）
- `XResolution`/`YResolution`（偏移 0x12/0x14）：分辨率
- `BitsPerPixel`（偏移 0x19）：色深

QEMU 在 `0x118` 模式下典型返回值是：`PhysBasePtr=0xFD000000`，`pitch=4096`，`1024x768`，`32bpp`。这些信息会被提取并保存到固定地址，供后续内核代码使用。

⚠️ 注意：不是所有显卡都支持 `0x118` 模式。如果返回失败，说明这个模式不可用。在实际产品中应该枚举所有可用模式并选择一个合适的，但对于我们的教程，直接硬编码 `0x118` 就够了。

### VESA 视频模式设置

获取模式信息后，最后一步是设置视频模式。调用 INT 0x10 AX=0x4F02 BX=0x4118，这里 BX 的高位 bit 14（`0x4000`）是关键 —— 它启用线性帧缓冲。

没有 bit 14，我们得到的是银行切换模式，需要频繁调用 BIOS 切换显存窗口，非常麻烦。线性模式下，整个帧缓冲是一段连续的物理地址，直接读写就行。比如要设置 (x, y) 像素的值，只需要计算 `PhysBasePtr + y * pitch + x * 4`（每个像素 4 字节），然后写入颜色值。

设置模式后屏幕会清空，之前的文本输出会消失。这是正常的，因为显卡从文本模式切换到了图形模式。这时候传统的 `printf` 和 BIOS 输出全都失效了，我们必须依靠其他方式来验证程序是否正确运行。

### 帧缓冲信息保存

VBE 调用返回的信息散布在 ModeInfoBlock 的各个偏移位置，不方便直接使用。我们把关键信息打包成一个 16 字节的结构，保存到固定地址 `0x6400`。结构布局如下：

- 偏移 0-7：物理地址（8 字节，虽然实际只用 4 字节）
- 偏移 8-11：pitch（4 字节）
- 偏移 12-13：宽度（2 字节）
- 偏移 14-15：高度（2 字节）

后续内核代码可以从 `0x6400` 读取这些信息，而不需要了解 ModeInfoBlock 的具体布局。这种抽象降低了模块间的耦合。

保存过程使用了段覆盖前缀。源数据在 ES:DI（`0x6200:0`），目标在 GS（`0x640`）。`movl %es:MODE_PHYS_BASE_PTR(%di), %eax` 从 ES 段读取，`movl %eax, %gs:FB_INFO_PHYS_ADDR(%di)` 写到 GS 段。这里必须显式指定段，否则编译器默认使用 DS。

### 在 GDB 中验证 framebuffer

现在问题来了：一旦进入图形模式，我们怎么知道程序是否正确运行？屏幕黑了，文本输出消失了，BIOS 调用也没用了。这时候 GDB 就成了救命稻草。

首先检查我们保存的 framebuffer 信息：

```gdb
x/2gx 0x6400
```

输出类似：
```
0x6400: 0x00000000fd000000  0x0300040000000c00
```

解析一下：第一个 64 位数 `0xfd000000` 是 framebuffer 的物理地址。第二个数包含了其他信息：pitch=3072（0xc00），width=1024（0x400），height=768（0x300）。计算一下：`3072 / 1024 = 3`，每像素 3 字节，说明是 RGB888（24-bit）格式。等等，不是 32-bit 吗？实际上 QEMU 在这个模式下可能返回 24-bit 或 32-bit，需要根据实际 pitch 来判断。

最直接的验证方法是直接写 framebuffer。在 GDB 中执行：

```gdb
set {int}0xfd000000 = 0x00FF0000
```

这会在左上角写入一个红色像素。但你可能会发现看不太清，因为只写了一个像素，人眼难以察觉。更好的方法是画一条红线：

```gdb
set $i = 0
while $i < 200
  set {int}(0xfd000000 + $i*4) = 0x00FF0000
  set $i = $i + 1
end
```

或者画一个矩形（推荐）：

```gdb
set $y = 0
while $y < 100
  set $x = 0
  while $x < 100
    set {int}(0xfd000000 + ($y * 1024 + $x) * 4) = 0x00FF0000
    set $x = $x + 1
  end
  set $y = $y + 1
end
```

如果左上角出现一个红色方块，恭喜！framebuffer 地址是对的，模式切换成功了。这时候你会发现，在没有文本输出的世界里，framebuffer 就是你的"调试打印函数"。

⚠️ 注意：如果在实模式下（`.code16`）直接写 framebuffer，可能会访问失败或行为不稳定。最好在进入保护模式后再操作 framebuffer。或者用 GDB 来写，因为 GDB 不受实模式限制。

## 设计决策深度分析

### 决策：为什么使用两阶段引导（MBR + Stage2）

**问题**：MBR 只有 512 字节，减去签名（2 字节）和可能的分区表（64 字节），实际可用空间不到 450 字节。这点空间塞个字符串输出和磁盘读取就差不多了。我们需要更大的空间来做硬件初始化。

**本项目的做法**：
MBR 只负责三件事：初始化环境、打印启动消息、加载 Stage2。Stage2 负责所有硬件相关的初始化（A20、VESA）。MBR 加载 Stage2 到 `0x8000`，然后跳转过去执行。

**备选方案**：
1. 单阶段引导：把所有代码塞进 MBR，但这在 512 字节限制下几乎不可能实现任何复杂功能。
2. 使用 GRUB：跳过所有引导代码，直接用 GRUB 加载内核。但这样就学不到底层知识了。
3. 多阶段引导：MBR → Stage2 → Stage3 → ... 每个阶段负责一部分功能。这对复杂 OS 有意义，但对于我们的教程有点过度设计。

**为什么不选备选方案**：
单阶段引导空间不够；使用 GRUB 跳过了学习机会；多阶段引导增加了复杂度但没有带来实质性好处。两阶段引导是空间和复杂度的最佳平衡点。

**如果要扩展/改进，应该怎么做**：
可以增加 Stage3 负责更复杂的初始化（如 ACPI 表枚举、SMP 启动），或者让 MBR 支持 FAT 文件系统，从文件加载 Stage2 而不是固定 LBA。这样就可以不重新编译 MBR 就能更新 Stage2。

### 决策：为什么使用 AT&T 语法而非 Intel 语法

**问题**：
x86 汇编有两种主流语法：Intel 语法（NASM、MASM 使用）和 AT&T 语法（GNU AS 使用）。选择哪种会影响代码可读性和工具链兼容性。

**本项目的做法**：
使用 GNU AS 的 AT&T 语法，因为 GCC 工具链原生支持。操作数顺序是 `src, dest`，寄存器有 `%` 前缀，立即数有 `$` 前缀。

**备选方案**：
1. 使用 NASM 和 Intel 语法：语法更直观，文档更多，但需要额外安装 NASM。
2. 使用内联汇编：在 C 代码中嵌入汇编，但引导阶段没有 C 运行时环境。

**为什么不选备选方案**：
GNU AS 已经是 GCC 工具链的一部分，不需要额外安装。虽然 AT&T 语法一开始不直观，但习惯后问题不大。而且 `.code16` 指令在 GNU AS 中支持良好，NASM 对 16 位代码的支持反而需要额外选项。

**如果要扩展/改进，应该怎么做**：
如果团队对 Intel 语法更熟悉，可以考虑用 NASM 重写引导代码。或者使用 `objcopy` 把两种语法的编译结果链接在一起，但这会增加构建复杂度。

### 决策：为什么选择 0x118 模式（1024x768x32）作为默认图形模式

**问题**：
VBE 支持多种图形模式，我们需要选择一个合适的作为默认模式。选择时要考虑分辨率、色深、兼容性。

**本项目的做法**：
硬编码使用 `0x118` 模式（1024x768x32 线性帧缓冲）。这个模式在 QEMU 和大多数现代显卡上都支持，分辨率适中，32 位色深可以显示完整的 ARGB 颜色。

**备选方案**：
1. 枚举所有可用模式，选择最佳的。这样更灵活，但代码更复杂。
2. 使用标准 VGA 模式（如 0x101，640x480x8）。兼容性更好，但分辨率和色深都不够。
3. 让用户通过配置文件指定模式。更灵活，但引导阶段需要解析文件系统。

**为什么不选备选方案**：
枚举所有模式需要很多代码，在引导阶段不值得。VGA 模式太老旧。配置文件需要文件系统支持，这在早期引导阶段太复杂。

**如果要扩展/改进，应该怎么做**：
可以枚举模式并选择第一个支持的 32 位色深模式，或者优先选择更高分辨率（如 1280x1024）。或者实现一个简单的配置解析器，从固定位置读取模式号。

## 常见变体与扩展方向

1. **从 FAT 文件系统加载 Stage2**（⭐⭐）
   修改 MBR 使其支持 FAT12/FAT32 文件系统，从文件加载 Stage2 而不是固定 LBA。这样更新 Stage2 不需要重新编译 MBR。需要实现简单的 FAT 解析，包括 MBR 解析、FAT 表遍历、簇链读取。

2. **实现命令行界面**（⭐⭐⭐）
   在 Stage2 中实现一个简单的命令行界面，支持 `help`、`mode`、`boot` 等命令。需要实现输入缓冲、命令解析、参数处理。可以参考 GRUB 的命令行设计。

3. **支持多种图形模式**（⭐⭐）
   枚举所有可用模式，让用户选择或者自动选择最佳模式。需要遍历模式列表（从 VbeInfoBlock），查询每个模式的信息，筛选出符合条件的（如 32 位色深、线性帧缓冲）。

4. **显示启动画面**（⭐）
   在切换到图形模式后，在屏幕上绘制一个简单的启动画面（如 Cinux logo）。需要实现简单的图形绘制函数（画点、画线、画矩形），以及图像解码器（如 BMP）。

5. **串口调试输出**（⭐）
   除了屏幕输出，同时通过串口发送调试信息。这样可以在 QEMU 的 `-serial stdio` 模式下查看日志。需要初始化串口（设置波特率、数据位等），实现 `print_string_serial` 函数。
